# Validation Patterns

这份文件回答的不是“怎么打一个漂亮分数”，而是“在不同约束下，什么证据算诚实”。

## 1. Replay

适用于：

- 能拿代表性 prompt 做 before / after 对比
- Skill 输出主要是文本、结构化建议、流程判断
- 成本可控

怎么用：

- 对同一组 prompt，比较 baseline 和 candidate
- 看差异是否真的贴着失败模式
- 不要只比较“更长 / 更完整”

结论强度：

- 中到强

## 2. Dry-Run

适用于：

- 没法真实 replay
- 任务成本过高
- 当前环境受限

怎么用：

- 明确写 `eval_mode = dry_run`
- 用代表性 prompt 推演执行路径
- 判断流程是否更合理、边界是否更稳

注意：

- dry-run 只能支持保守结论
- 它更适合排除明显坏改动，不适合宣布“已经证明更好”

结论强度：

- 弱到中

## 3. Human Review

适用于：

- 风格、写作、人格、审美判断占比很高
- 任务存在强主观偏好
- 真实用户目标很难被 proxy 覆盖

怎么用：

- 不追求单一总分
- 重点看是否更贴近 brief、voice、边界和用户意图
- 可以用 prompt replay 做辅助，但最终判断交给人工

结论强度：

- 取决于 review 质量，但通常比假装“客观评分”更诚实

## 4. 分层验证原则

复杂任务不要只看一个指标。优先按三层看：

1. 合同型指标：有没有跑偏
2. 代理型指标：有没有更贴近目标输出
3. 真实反馈：用户是否真的更想要

对于写作、风格、人格类 Skill，第三层通常不能被离线分数替代。
